# 面向同区域跨孤岛多场站极端天气联邦少样本预测的严格版问题定义与实施顺序

## 0. 文档定位

本文档是当前 `WPF-under-extreme-weather-main` 联邦化工作的**严格版架构决策说明**，目标不是提出一个“听起来完整”的一次性大重构，而是基于现有代码、原始 WPF 论文、`F2L` 论文与 `pFedFSL` 论文，给出：

1. 哪些问题已经被证实是真问题；
2. 哪些机制可以保留为主线；
3. 哪些机制只能降级为候选方案；
4. 真正可执行的实施顺序是什么。

本文档默认的目标场景是：

- 多场站数据属于不同主体，不能做中心化拼接；
- 联邦通信只能上传共享模块参数或其更新；
- 极端天气样本稀缺，且各场站极端天气覆盖不一致；
- 需要兼顾预测精度、个性化能力、通信代价与隐私风险。

---

## 1. 已确认事实

### 1.1 当前代码并不等于“严格联邦”

当前主线代码虽然打开了 `USE_FEDERATION=True`，但其联邦化仍然是**不完整的**：

- 预训练阶段采用了多场站梯度平均；
- 元训练阶段仍然把所有场站任务放入**全局任务池**中统一采样；
- few-shot 适配仍然在单进程内顺序访问所有场站数据。

因此，当前实现更准确的表述应是：

> “在单机环境下模拟多场站参与的部分联邦训练流程”，而不是“严格数据孤岛联邦”。

这意味着，后续所有重构都必须首先解决“联邦边界不严格”的问题，而不是直接叠加更复杂的论文机制。

### 1.2 原始 WPF 论文的关键方法边界

原始 `Wind Power Forecasting Under Extreme Weather: a Novel Few-Shot Learning Architecture` 论文明确包含以下核心设定：

- 常规天气 source domain 先做 `pre-train`；
- source domain 数据通过改进 `K-means` 做 `task clustering`；
- meta-training 采用 support/query 跨任务更新；
- `LWP` 的作用是冻结主干、只更新轻量参数与 base learner；
- `CDRM` 依赖 mini-batch 间梯度约束；
- step-11 fine-tuning 使用 target few-shot 数据做适配。

这些设定在单场站、单数据域论文环境下是自洽的，但直接搬到真实联邦数据孤岛场景时并不天然成立。

### 1.3 F2L 论文的真实贡献边界

`Federated Few-shot Learning (F2L)` 真正支持的是：

- `client-model / server-model` 的**解耦式元学习框架**；
- 从本地 client-model 向 server-model 的**互信息最大化传递**；
- 从 server-model 回到 client 的**partial knowledge distillation (PKD)**；
- server-model 仍通过聚合更新。

F2L **不直接提供**以下能力：

- 去中心化兄弟节点路由；
- 基于兄弟模型本地打分的 softmax 动态聚合；
- 面向时序风电场景的 LWP/TCN 专用实现。

### 1.4 pFedFSL 论文的真实贡献边界

`Personalized Federated Few-Shot Learning (pFedFSL)` 真正支持的是：

- 客户端基于本地性能，对收到的多个模型计算聚合权重；
- 服务器自适应选择“哪些客户端模型发给谁”；
- 通过相似客户端协作增强 few-shot 个性化。

pFedFSL 的思想与“去中心化性能路由”高度相关，但它不是 F2L，也不是原始 WPF 的直接延伸。

## 2. 严格定义下的致命问题

### 致命问题 0：当前联邦边界不严格，主线结果不能代表真实数据孤岛联邦

这是当前工作的最高优先级问题。

如果元训练仍然使用全局任务池统一采样，那么：

- 场站之间的数据仍在同一训练进程内被显式访问；
- “服务器只看共享参数、不看本地任务样本”的联邦前提并未建立；
- 后续关于隐私、通信与真实联邦可部署性的讨论都会失去基础。

**结论：**
在进入任何高级机制之前，必须先把训练流程改造成严格的“共享参数联邦 + 本地任务生成 + 本地适配”结构。

---

### 致命问题 1：原始 CDRM 形式不适合直接作为真实联邦主线损失

原始 WPF 论文中的 CDRM 依赖 mini-batch 间的梯度约束，本质上是围绕：

- 二阶梯度；
- batch 级别的梯度相关性；
- 多任务一致性约束

来构造的。

在单机场景下，这种形式是可计算的；但在真实跨主体联邦场景下，它存在三个问题：

1. **实现边界不清**：若要精确保留跨任务/跨客户端的约束，需要额外设计安全协议或近似形式；
2. **通信与计算代价高**：二阶梯度和 batch 级别约束会显著放大计算与通信复杂度；
3. **隐私风险更敏感**：更细粒度的梯度交互会提升攻击面。

这里必须严格澄清：

- 不能简单写成“CDRM 在联邦下必然要求实时上传原始 mini-batch 梯度”；
- 也不能写成“DLG 一定可以 100% 完美恢复风速和功率曲线”。

更严谨的表述应是：

> 原始 CDRM 的精确形式对真实联邦不友好，会引入额外的安全协议需求、较高通信/计算成本，以及更敏感的梯度泄露风险，因此不应直接作为第一阶段主线机制。

---

### 致命问题 2：原始 source-domain task clustering 在多数据孤岛场景下缺乏跨站语义对齐

WPF 原法依赖对 source-domain 常规天气数据做 `K-means` 聚类，以形成 task。

在单场站场景中，这样做没有问题；但在多场站数据孤岛场景中：

- 场站 A 的 `Task-1` 与场站 B 的 `Task-1` 不具备天然同义性；
- 聚类边界受本地风场、地形、机型、局地天气统计特征共同影响；
- 即使聚类编号相同，也不代表物理机制一致。

因此，原论文中“任务”这一中间结构不能直接被看作跨场站可对齐的联邦对象。

**结论：**
跨站联邦中的共享对象不能默认是 “task id”，而应优先是：

- 共享主干参数；
- 共享表征能力；
- 或通过验证机制动态确定的可迁移知识。

---

### 致命问题 3：LWP 与 base learner 不应作为全局平均对象

LWP 在原论文中本身就承担了“针对 shifted domain 做局部适配”的角色。

在多场站场景下，LWP 和 base learner 更进一步承载了：

- 微地形差异；
- 机组控制策略差异；
- 场站功率曲线差异；
- 局地极端天气响应差异。

如果服务器直接把这些参数做 FedAvg，会出现两个问题：

1. 把本应属于本地个性化的参数做了跨站平均；
2. 生成对任何站都不够好的“折中参数”。

**结论：**
严格主线中，`LWP + base learner + 极端天气 few-shot 头部` 必须视为本地个性化模块，不应作为默认聚合对象。

---

### 致命问题 4：仅依赖少量横向客户端协作，难以覆盖罕见极端天气

即使改成个性化联邦，仍然会遇到一个现实问题：

- 客户端总数只有 3-4 个；
- 极端天气分布极不均衡；
- 某一站当前遭遇的极端天气，其他站未必见过。

这意味着仅靠“当前横向兄弟客户端”来帮助目标站点，覆盖能力是有限的。

**结论：**
横向协作可以作为增强手段，但不能被假设为足以解决所有罕见极端天气。任何基于兄弟站点的协作机制，都必须被设计为“增益项”，而不是唯一支柱。

---

## 3. 保留为主线的内容

下列内容在严格版中**保留为主线**：

### 3.1 保留 `TCN backbone + LWP + base learner` 的基本结构

理由：

- 这是当前代码、原始论文与已有实验资产的共同基础；
- LWP 的“冻结主干、轻量适配 shifted domain”思想仍然有价值；
- 彻底换骨干会让问题从“联邦少样本极端天气预测”退化成“重新造一个全新模型”。

### 3.2 保留 step-11 式 few-shot 本地适配

理由：

- 原始论文中 target domain few-shot 适配是核心闭环；
- 当前代码已经对齐到 “fine-tune 使用 experience loss / MSE” 的论文口径；
- 在联邦场景中，本地极端天气适配反而更合理，因为它天然符合数据不出域。

### 3.3 保留“共享模块 + 本地模块”二层分解

但严格版中要明确：

- **共享模块**：TCN backbone 或其等价共享表征模块；
- **本地模块**：LWP、base learner、极端天气 few-shot 适配参数。

这不是候选方向，而是严格主线。

### 3.4 保留“先常规天气学习，再极端天气少样本适配”的大顺序

严格版不否定原论文的整体训练逻辑，只是否定其原始实现形式直接搬到联邦场景。

因此主线仍然是：

1. 常规天气阶段学习共享知识；
2. 极端天气阶段做少样本本地适配；
3. 在此基础上再讨论是否需要更复杂的联邦知识增强机制。

## 4. 降级为候选方案的内容

下列内容**不再作为第一阶段默认主线**，而是降级为候选方案或后续消融方向。

### 候选方案 A：F2L 风格的双模型解耦 + MI + PKD

该方案保留，但降级为**第二阶段增强模块**。

严格定义如下：

- `client-model` 负责本地元知识；
- `server-model` 负责共享表征；
- `MI` 用于 client-model 到 server-model 的 local-to-global 知识传递；
- `PKD` 用于 server-model 到 client 的 global-to-local 知识蒸馏。

必须明确：

- 这里的 MI 不是“输入特征和隐层表示之间的 MI 最大化”；
- 而是 `client-model` 与 `server-model` 表征之间的知识传递。

因此，旧文档中“借鉴 F2L 就直接在本地做输入-隐层 MI”这一写法不再保留。

### 候选方案 B：pFedFSL 风格的本地性能加权路由

该方案保留，但降级为**第三阶段 personalization/routing 模块**。

严格定义如下：

- 客户端收到若干兄弟模型；
- 用本地数据评估这些模型在本地的效果；
- 用 softmax 或类似机制生成加权聚合；
- 服务器根据历史权重决定下一轮把哪些模型发给谁。

必须明确：

- 这是 `pFedFSL` 风格的思路；
- 不是 F2L 的原始方法；
- 也不是当前第一阶段 baseline 的必要条件。

### 候选方案 C：历史 Teacher + PKD

该方案保留，但降级为**研究型候选方案**，不能直接写成既定落地路线。

在以下问题未定义清楚前，不进入主线：

1. Teacher 是按轮次、按季度还是按版本更新；
2. Teacher 是 backbone teacher 还是 full model teacher；
3. 蒸馏对齐 feature、logit 还是二者混合；
4. Teacher 的训练数据窗口如何定义；
5. Teacher 是否会把历史偏差放大。

因此，旧文档中“全面转向历史大模型蒸馏”这一表述降级为候选研究方向。

### 候选方案 D：原始 CDRM 的联邦近似版本

若未来仍希望保留跨任务不变性思想，可以探索：

- 仅本地客户端内部使用的近似正则；
- 不需要跨客户端细粒度梯度交互的替代形式；
- 或 F2L / distillation 风格的表征层一致性约束。

但在第一阶段，不再以“精确保留原始 CDRM”作为目标。

## 5. 明确删除或禁止的表述

以下说法在严格版中**明确删除**：

1. “原始 CDRM 在联邦下必然要求实时上传单一 mini-batch 原始梯度。”
2. “DLG 攻击可以 100% 完美恢复真实风速和功率曲线。”
3. “F2L 直接支持去中心化兄弟模型性能路由。”
4. “只要改成 MI，就自然能提炼跨域不变特征。”
5. “全面废除任务机制后，历史 Teacher 一定优于横向协作。”

这些表述要么过度绝对化，要么混淆了不同论文的方法边界。

## 6. 真正的实施顺序

严格版采用分阶段实施，而不是一次性大重构。

### 第一阶段：先建立严格联邦 baseline

目标：
先得到一个真正符合数据孤岛约束、结构清楚、可复现实验的基线系统。

执行内容：

1. 重新划定联邦边界。
   - 服务器只能接收共享模块参数或更新；
   - 本地任务构造、few-shot 适配和个性化参数留在客户端。
2. 将共享模块限定为 `TCN backbone`。
3. 将 `LWP + base learner + 极端天气 few-shot 参数` 固定为本地模块。
4. 去掉元训练中对全局任务池的直接拼接访问。
5. 先跑“联邦常规天气预训练 + 本地极端天气 few-shot 适配”。

第一阶段的成功标准：

- 训练流程不再直接中心化访问各场站任务样本；
- 结果可以和当前“伪联邦版”直接对比；
- 能明确看到“共享骨干 + 本地适配器”是否优于“共享一切”。

### 第二阶段：重构联邦 meta-learning，而不是直接套原始单机场景 meta-training

目标：
把 meta-learning 从“单机全局任务池”改成“每客户端本地任务、服务器只聚合共享知识”的形式。

执行内容：

1. 每个客户端本地构造 episodic task；
2. 支持/查询集仅由本地常规天气数据构造；
3. 服务器只聚合共享 backbone 的可共享部分；
4. 本地保留 LWP / head 的更新轨迹。

这一阶段解决的是“task clustering 在联邦下如何存在”的问题，而不是先追求 fancy loss。

### 第三阶段：引入 F2L 风格的 MI/KD 作为增强模块

目标：
在严格 baseline 稳定后，引入更强的 global-to-local / local-to-global 知识传递。

执行内容：

1. 引入 `client-model / server-model` 双模型结构；
2. 用 MI 做本地到全局的知识迁移；
3. 用 PKD 做全局到本地的知识蒸馏；
4. 做无 MI、无 KD、双者同时存在的消融。

### 第四阶段：将 pFedFSL 风格的性能路由作为个性化增强模块

目标：
解决“并非所有兄弟站点都值得协作”的问题。

执行内容：

1. 客户端对候选兄弟模型做本地验证；
2. 基于本地损失或验证表现做 softmax 加权；
3. 服务器维护“哪些客户端对哪些客户端更有帮助”的路由矩阵；
4. 比较固定邻接、随机发送、自适应发送三种策略。

### 第五阶段：再评估是否需要历史 Teacher / PKD

只有在前四阶段完成后，才讨论：

- 是否真的存在“横向协作覆盖不足”的稳定证据；
- 历史 Teacher 是否比兄弟模型协作更稳；
- Teacher 带来的额外系统复杂度是否值得。

在没有这些证据之前，历史 Teacher 只作为候选研究方向，不进入主线。

## 7. 验证与消融要求

严格版必须把验证顺序写清楚，否则架构升级无法判断收益来源。

### 7.1 必做对比

1. 当前伪联邦主线 vs 严格联邦 baseline；
2. 共享 backbone + 本地 LWP/head vs 共享所有参数；
3. 无 meta-learning vs 本地 episodic meta-learning；
4. 无 MI/KD vs MI only vs KD only vs MI+KD；
5. 无路由 vs 随机发送 vs 自适应路由。

### 7.2 必测指标

1. 预测指标：`nMAE / nRMSE / WD / Rp<0.05`；
2. 系统指标：每轮通信量、总训练时长、本地 few-shot 时长；
3. 个性化指标：各站点之间的性能方差；
4. 稀有极端天气指标：在覆盖最弱天气类型上的表现。

### 7.3 必须记录的系统约束

1. 服务器看到的参数范围；
2. 客户端保留的私有参数范围；
3. 每轮发送对象；
4. 是否存在 batch 级额外交互；
5. 是否存在需要共享中间表征的步骤。

---

## 8. 最终架构决策摘要

### 8.1 保留为主线

- `TCN backbone + LWP + base learner` 基本结构；
- 本地 few-shot 适配；
- “共享模块 + 本地模块”二层分解；
- “先常规天气共享学习，再极端天气本地适配”的总体顺序。

### 8.2 降级为候选方案

- F2L 风格 `dual-model + MI + PKD`；
- pFedFSL 风格本地性能路由；
- 历史 Teacher / PKD；
- 联邦近似版 CDRM。

### 8.3 当前真正优先级

> 先把“伪联邦”纠正为“严格联邦 baseline”，再谈 MI、KD、路由和历史 Teacher。

如果第一步没有完成，那么后续所有复杂机制都只是在不严格边界上继续堆复杂度。
